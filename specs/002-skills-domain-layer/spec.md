# Spec: Skills Domain Layer

> Upgrade the 6 speckit-* skills to inject domain context, use shared rules, support sub-agent dispatch, and include Safety & Fallback.

## P1 — MVP

### Story 1: Shared Rules Foundation
**As a** skill author, **I want** shared rule files that multiple skills can reference, **so that** spec-writing, design, and task-generation follow consistent patterns.

**Acceptance:**
- Given a `skills/rules/` directory with 3 files: `spec-writing.md`, `design-principles.md`, `tasks-generation.md`
- When a skill loads, it can reference shared rules via `metadata.shared-rules`
- Then all skills use consistent language for boundaries, acceptance criteria, and phase structure

### Story 2: Domain Context Injection
**As a** skill, **I want** to read steering context (product.md, tech.md, structure.md) and domain-map.md before generating artifacts, **so that** output is grounded in real project decisions.

**Acceptance:**
- Given a project with steering documents and domain-map.md
- When speckit-spec-writer or speckit-plan-engineer runs
- Then it reads and references these documents before generating
- Given no steering documents exist
- When the skill runs
- Then it gracefully skips (no crash)

### Story 3: Conversational Output
**As a** user, **I want** skills to propose structure before writing files, **so that** I can confirm direction before files are created.

**Acceptance:**
- Given a skill is invoked
- When it has enough context to propose
- Then it presents a proposal to the user and waits for confirmation
- Only after confirmation does it write files to disk

### Story 4: Safety & Fallback
**As a** user, **I want** each skill to document what happens when things go wrong, **so that** I know how to recover without restarting.

**Acceptance:**
- Given any skill SKILL.md
- Then it includes a `## Safety & Fallback` section
- With at least 3 concrete error scenarios and recovery actions
- With no generic "ask the user" fallbacks — each must be specific

### Story 5: Boundary Annotations
**As a** developer, **I want** tasks to carry `_Boundary:_` and `_Depends:_` annotations, **so that** task boundaries are explicit and reviewable.

**Acceptance:**
- Given a tasks.md generated by speckit-task-decomposer
- Then every parallel task `[P]` has a `_Boundary:_` annotation
- Then every non-obvious dependency has `_Depends:_` annotation
- Then the reviewer can check for boundary violations

## P2 — Important

### Story 6: Sub-Agent Dispatch for Research
**As a** plan-engineer skill, **I want** to dispatch sub-agents for parallel research (dependencies, alternatives, codebase analysis), **so that** main context stays clean.

**Acceptance:**
- Given a complex technology choice
- When speckit-plan-engineer runs
- Then it can dispatch sub-agents for parallel research
- Then it synthesizes findings before writing the plan

### Story 7: Phase Tracking via spec.json
**As a** tool, **I want** each feature to have a spec.json with phase metadata, **so that** tools can track progress programmatically.

**Acceptance:**
- Given a feature directory
- Then it contains `spec.json` with `phase`, `approvals`, and `updated_at`
- When a skill completes an artifact
- Then it updates the phase in spec.json
